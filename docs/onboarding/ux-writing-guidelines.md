## Tone By Surface

### Public Blog And Product Pages

- 기본 톤은 친근한 `~요`를 사용한다.
- 사용자가 읽는 블로그, 릴리즈 노트, 댓글, 공유, 로그인 문구는 부드럽고 짧게 쓴다.
- 큰 제목, 버튼, 메뉴, 라벨에는 마침표를 쓰지 않는다.

### Backoffice

- 일반 상태, 성공 토스트, 빈 상태는 간결한 `~요`를 사용한다.
- 삭제, 복구 불가, 권한, 감사 로그, 보안처럼 사용자가 조심해야 하는 상황은 `~습니다`를 사용할 수 있다.
- 운영자가 정확한 값을 복사하거나 식별해야 할 때만 기술 용어를 그대로 둔다. 그 외에는 기능의 목적을 설명한다.

## Core Rules

### 쉬운 말을 쓴다

사용자가 바로 이해할 수 있는 말을 우선한다. 기술 용어, 한자어, 내부 구현명은 필요한 경우에만 쓴다.

- 권장: `요청을 처리하지 못했어요. 잠시 후 다시 시도해주세요.`
- 지양: `unknown error`, `서버 에러`, `토큰 암복호화 실패`

### 오류 메시지는 다음 행동을 알려준다

오류 메시지는 상태 보고에서 끝내지 않는다. 사용자가 직접 할 수 있는 행동을 함께 쓴다.

- 다시 시도할 수 있는 문제: `잠시 후 다시 시도해주세요.`
- 입력 문제: `입력한 값을 확인해주세요.`
- 권한 문제: `필요한 권한이 있다면 운영진에게 문의해주세요.`

### 존칭어는 최소화한다

`하실 수 있어요`, `계신가요`, `사용하시면 됩니다`처럼 과한 존칭은 줄인다. 사용자 이름이나 닉네임을 직접 부르는 문구는 소리내어 읽었을 때 어색하지 않은지 확인한다.

- 권장: `로그인하면 이미지 업로드를 사용할 수 있어요.`
- 지양: `로그인하시면 이미지 업로드를 사용하실 수 있습니다.`

### 숫자는 아라비아 숫자로 쓴다

수량, 횟수, 제한값은 숫자로 쓴다.

- 권장: `선택지는 2개 이상 입력해주세요.`
- 지양: `선택지는 두 개 이상 입력해주세요.`

### 축약어, 은어, 유행어를 줄인다

팀 내부에서 쓰는 줄임말이 사용자에게도 명확한지 확인한다. 공개 화면에서는 특히 줄임말보다 풀어쓴 표현을 우선한다.

- 권장: `동기화`
- 지양: `sync`

### 기능명보다 목적을 말한다

버튼과 안내 문구는 내부 기능 이름보다 사용자가 하려는 일을 보여준다.

- 권장: `Figma 파일 연결하기`
- 지양: `OAuth 위임`

### 긍정문을 우선한다

같은 의미라면 부정문보다 가능한 행동과 조건을 먼저 말한다.

- 권장: `지부 공지를 작성하려면 기수를 먼저 선택해주세요.`
- 지양: `기수를 선택하지 않은 상태에서는 지부 공지를 작성할 수 없습니다.`

### 능동문을 우선한다

문장이 자연스럽다면 사용자의 행동이나 시스템의 결과를 능동적으로 쓴다. 단, 자동 발송 알림이나 위험 안내처럼 무게가 필요한 상황에서는 더 단정적인 문장을 사용할 수 있다.

- 권장: `상벌점을 부여했어요.`
- 지양: `상벌점이 부여되었습니다.`

### 사용자 행위를 중심에 둔다

`처리`, `완료`, `상태 변경` 같은 데이터 변화보다 사용자가 한 일을 말한다.

- 권장: `댓글을 수정했어요.`
- 지양: `댓글이 수정되었습니다.`

### 한 문장에는 하나의 목적만 담는다

설명이 길어지면 문장을 나눈다. 특히 오류, 권한, 위험 안내는 원인과 다음 행동을 분리하면 읽기 쉽다.

- 권장: `Figma 연결이 만료됐어요. 다시 연결해주세요.`
- 지양: `Figma access token 갱신에 실패했으므로 재위임이 필요합니다.`

### 느낌표는 아껴 쓴다

느낌표는 환영, 축하, 강한 주의가 필요한 순간에만 쓴다. 빈 상태나 반복되는 토스트에는 쓰지 않는다.

### 띄어쓰기는 의미에 맞춘다

기능명, 서비스명, 상태값 자체를 가리킬 때는 제품에서 정한 표기를 따른다. 일반 문장에서는 자연스러운 띄어쓰기를 우선한다.

- 상태값: `모집중`, `모집완료`처럼 디자인/도메인에서 정한 표기 사용
- 일반 문장: `모집 중일 때는 수정할 수 없어요.`

### 마침표는 문맥에 맞춘다

평서문과 명령문의 끝에는 마침표를 쓸 수 있다. 단, 아래 문구에는 마침표를 쓰지 않는다.

- 메인 타이틀
- 큰 글씨로 들어가는 제목
- 버튼, 메뉴, 탭, 라벨
- 빈 상태 제목

## Review Checklist

문구를 추가하거나 수정할 때 아래 질문을 확인한다.

- 사용자가 바로 이해할 수 있는 말인가?
- 오류 메시지에 다음 행동이 있는가?
- 기능명보다 사용자의 목적을 말하는가?
- `완료되었습니다`, `처리되었습니다`, `오류가 발생했습니다`처럼 상태 보고에 머물지 않는가?
- 버튼, 라벨, 큰 제목에 불필요한 마침표가 없는가?
- 느낌표가 꼭 필요한가?
- 기술 용어를 그대로 보여줄 이유가 있는가?
