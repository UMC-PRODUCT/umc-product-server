# 프로젝트 권한 정책

프로젝트, 프로젝트 지원용 폼, 프로젝트 지원서, 매칭 차수 도메인의 권한 정책을 정리합니다.

## 공통 원칙

| 항목 | 정책 |
| --- | --- |
| 참조 관계 삭제 제한 | 다른 도메인 또는 엔티티에서 참조 중인 데이터는 삭제할 수 없습니다. |
| 조회 공개 범위 | 별도 제한이 명시되지 않은 조회 API는 도메인별 정책을 따릅니다. |
| 운영진 권한 범위 | 지부장은 본인 지부 범위, 회장은 본인 학교 범위, 중앙운영사무국 총괄단 이상은 전체 범위를 기준으로 판단합니다. |
| 프로젝트 소유자 | `Project PO`는 프로젝트 생성, 수정, 삭제 및 관련 폼 관리의 핵심 권한자입니다. |

## 역할 기준

| 역할 | 권한 범위 |
| --- | --- |
| Plan 챌린저 | 본인이 `Project PO`인 프로젝트를 생성할 수 있습니다. |
| 회장 | 본인 학교 인원을 `Project PO`로 지정할 수 있습니다. |
| 지부장 | 본인 지부 소속 학교 인원을 `Project PO`로 지정할 수 있습니다. |
| 중앙운영사무국 총괄단 이상 | 학교 및 지부 제한 없이 프로젝트와 매칭 차수를 관리할 수 있습니다. |
| Project PO | 본인이 담당하는 프로젝트, 프로젝트 지원용 폼, 지원서 합불 결정을 관리할 수 있습니다. |
| Sub-PO | 제출된 지원서 조회 권한을 가집니다. |
| 지원자 | 본인이 생성한 지원서를 조회, 수정, 삭제할 수 있습니다. |

| 도메인 | 생성 | 조회 | 수정 | 삭제 | 비고 |
| --- | --- | --- | --- | --- | --- |
| 프로젝트 | 역할별 `Project PO` 지정 범위 내 가능 | Public API | `Project PO`만 가능 | `Project PO`만 가능 | 프로젝트 자체 조회 및 매칭 현황 조회는 공개합니다. |
| 프로젝트 지원용 폼 | `Project PO`만 가능 | 정책 확인 필요 | `Project PO`만 가능 | `Project PO`만 가능 | 지원서 생성 가능 여부의 기준이 됩니다. |
| 프로젝트 지원서 | 폼 정책에서 허용한 파트의 챌린저만 가능 | 지원자 본인, `Project PO`, `Sub-PO`, 지부장급 이상 운영진 | 생성자만 가능 | 생성자만 가능 | 합불 결정은 `Project PO`만 가능합니다. |
| 매칭 차수 | 지부장 또는 중앙운영사무국 총괄단 이상 가능 | 정책 확인 필요 | 지부장 또는 중앙운영사무국 총괄단 이상 가능 | 지부장 또는 중앙운영사무국 총괄단 이상 가능 | 지원서 등 참조 관계가 있으면 삭제할 수 없습니다. |

## 프로젝트 권한

### 프로젝트 생성

프로젝트 생성 시 `Project PO`로 지정할 수 있는 대상은 요청자의 역할과 소속 범위에 따라 달라집니다.

| 요청자 역할 | `Project PO` 지정 가능 범위 |
| --- | --- |
| Plan 챌린저 | 본인만 지정 가능 |
| 회장 | 본인 학교 인원만 지정 가능 |
| 지부장 | 본인 지부 소속 학교 인원만 지정 가능 |
| 중앙운영사무국 총괄단 이상 | 제한 없음 |

### 프로젝트 수정 및 삭제

| 행위 | 허용 대상 | 제한 사항 |
| --- | --- | --- |
| 수정 | `Project PO` | `Project PO`가 아닌 사용자는 수정할 수 없습니다. |
| 삭제 | `Project PO` | 참조 관계가 있는 경우 삭제할 수 없습니다. |

### 프로젝트 조회

| 조회 대상 | 공개 범위 | 비고 |
| --- | --- | --- |
| 프로젝트 자체 조회 | Public API | 인증 또는 역할 제한 없이 공개합니다. |
| 매칭 현황 조회 | Public API | `ProjectMember` 조회를 포함해 공개합니다. |

## 프로젝트 지원용 폼 권한

프로젝트 지원용 폼은 프로젝트 지원서 생성 가능 여부를 판단하는 정책 기준입니다.

| 행위 | 허용 대상 | 비고 |
| --- | --- | --- |
| 생성 | `Project PO` | 본인이 담당하는 프로젝트에 대해서만 가능합니다. |
| 수정 | `Project PO` | 지원 가능 파트, 차수 등 지원 정책 변경 시 영향을 받는 지원서 정책을 함께 검토해야 합니다. |
| 삭제 | `Project PO` | 참조 관계가 있는 경우 삭제할 수 없습니다. |

## 프로젝트 지원서 권한

### 지원서 생성

지원서 생성은 프로젝트 지원용 폼의 정책에서 허용한 파트의 챌린저에게만 허용됩니다.

| 조건 | 정책 |
| --- | --- |
| 지원 가능 파트 | 지원용 폼에서 제한한 파트에 포함된 챌린저만 지원할 수 있습니다. |
| 기수 내 기존 팀 소속 여부 | 이미 해당 기수에 소속된 팀이 있으면 중복 지원할 수 없습니다. |
| 동일 차수 중복 제출 | 이미 동일한 차수에 제출한 지원서가 있으면 중복 지원할 수 없습니다. |
| DB 제약 | 동일 차수 중복 제출은 Unique Key로도 제한됩니다. |

> 1차 매칭에서 팀이 결정된 경우 2차 매칭에 지원할 수 없습니다. 단, 1차에서 강제 배정되었을 수도 있으므로 팀 소속 여부 판단 로직은 배정 경로와 상태를 함께 고려해야 합니다.

### 지원서 조회

| 조회자 | 조회 가능 범위 |
| --- | --- |
| 지원자 본인 | 본인이 제출한 지원서 |
| `Project PO` | 본인 프로젝트에 제출된 지원서 |
| `Sub-PO` | 본인이 `Sub-PO`로 지정된 프로젝트에 제출된 지원서 |
| 지부장급 이상 운영진 | 권한 범위 내 지원서 |

### 지원서 수정 및 삭제

| 행위 | 허용 대상 | 제한 사항 |
| --- | --- | --- |
| 수정 | 지원서 생성자 | 본인이 생성한 지원서만 수정할 수 있습니다. |
| 삭제 | 지원서 생성자 | 본인이 생성한 지원서만 삭제할 수 있습니다. |

### 지원서 합불 결정

| 행위 | 허용 대상 | 제한 사항 |
| --- | --- | --- |
| 합격 처리 | `Project PO` | 운영진이 별도로 합불을 결정할 수 없습니다. |
| 불합격 처리 | `Project PO` | 운영진이 별도로 합불을 결정할 수 없습니다. |
| 지원 현황 조회 | `Project PO`, `Sub-PO`, 운영진 | 지원자 본인에게는 전체 지원 현황 조회 권한이 없습니다. |

### 매칭 차수 경과 시 자동 합불 처리

지원서의 매칭 차수가 경과했는데 `Project PO`가 합불을 결정하지 않은 경우, 최소 선발 규칙에 따라 자동 합불 처리를 수행해야 합니다.

| 항목 | 정책 |
| --- | --- |
| 자동 선발 기준 | 최소 선발 인원만 랜덤으로 합격 처리합니다. |
| 자동 탈락 기준 | 최소 선발 인원을 제외한 나머지 지원자는 불합격 처리합니다. |
| 규칙 차이 | 기획-디자인 매칭과 기획-개발 매칭의 최소 선발 규칙은 서로 다릅니다. |
| 참고 문서 | [10th UMC PLAN(팀장) 필독 공지](https://www.notion.so/10th-UMC-PLAN-33ab57f4596b81ae927ec25d0892849b?pvs=21)의 선발 필수 규칙을 확인해야 합니다. |

예시:

| TO | 지원자 수 | 최소 선발 인원 | 자동 처리 결과 |
| --- | --- | --- | --- |
| 4명 | 5명 | 2명 | 2명 랜덤 합격, 3명 불합격 |

## 매칭 차수 권한

### 매칭 차수 CRUD

| 요청자 역할 | 생성 | 조회 | 수정 | 삭제 | 범위 |
| --- | --- | --- | --- | --- | --- |
| 지부장 | 가능 | 가능 | 가능 | 가능 | 본인 지부 |
| 중앙운영사무국 총괄단 이상 | 가능 | 가능 | 가능 | 가능 | 전체 |
| 그 외 역할 | 불가 | 정책 확인 필요 | 불가 | 불가 | 해당 없음 |

### 매칭 차수 삭제 제한

| 조건 | 삭제 가능 여부 | 처리 기준 |
| --- | --- | --- |
| 참조 관계 없음 | 가능 | 요청자의 역할 및 범위 권한을 확인합니다. |
| 지원서 등 참조 관계 있음 | 불가 | 기존 참조 관계를 모두 제거한 뒤 삭제할 수 있습니다. |

## 구현 체크리스트

| 구분 | 확인 사항 |
| --- | --- |
| 프로젝트 생성 | 요청자의 역할에 따라 `Project PO` 지정 가능 범위를 검증합니다. |
| 프로젝트 수정/삭제 | 요청자가 해당 프로젝트의 `Project PO`인지 검증합니다. |
| 프로젝트 공개 조회 | 프로젝트 자체 조회와 `ProjectMember` 기반 매칭 현황 조회는 Public API로 둡니다. |
| 폼 관리 | 프로젝트 지원용 폼 생성, 수정, 삭제는 `Project PO`만 허용합니다. |
| 지원서 생성 | 폼의 파트 제한, 동일 기수 팀 소속 여부, 동일 차수 중복 제출 여부를 모두 검증합니다. |
| 지원서 조회 | 지원자 본인, `Project PO`, `Sub-PO`, 지부장급 이상 운영진만 허용합니다. |
| 지원서 수정/삭제 | 지원서 생성자만 허용합니다. |
| 합불 결정 | `Project PO`만 수행할 수 있도록 제한합니다. |
| 자동 합불 | 매칭 차수 경과 시 최소 선발 규칙에 따라 자동 합불 처리를 수행합니다. |
| 매칭 차수 관리 | 지부장은 본인 지부만, 중앙운영사무국 총괄단 이상은 전체 범위를 허용합니다. |
| 삭제 제한 | 참조 관계가 있는 리소스는 삭제를 거부합니다. |

## 미정 또는 추가 확인 필요 사항

| 항목 | 확인 필요 내용 |
| --- | --- |
| 프로젝트 지원용 폼 조회 권한 | 전체 공개인지, 프로젝트 관계자 또는 운영진으로 제한할지 결정이 필요합니다. |
| 매칭 차수 조회 권한 | Public API 여부 또는 운영진 제한 여부 결정이 필요합니다. |
| 운영진 범위 | "지부장급 이상 운영진"의 정확한 역할 enum 또는 권한 그룹 정의가 필요합니다. |
| 최소 선발 규칙 | 기획-디자인, 기획-개발 매칭별 최소 선발 산식과 예외 케이스를 코드화할 수 있도록 명세가 필요합니다. |
